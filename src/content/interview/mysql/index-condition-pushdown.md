## 核心定义

索引下推，英文是 Index Condition Pushdown，简称 ICP。它的作用是在使用索引扫描时，尽量把原本需要回表后再判断的条件，下推到存储引擎层提前过滤，从而减少回表次数。

## 没有索引下推时会怎样

假设联合索引是 `(name, age)`，SQL 条件里既有 `name` 又有 `age`：

1. 先通过索引定位到满足 `name` 的记录。
2. 再回表取整行数据。
3. 最后在 Server 层判断 `age` 条件。

如果命中了很多索引记录，就会产生大量无效回表。

## 有索引下推时会怎样

存储引擎在扫描索引记录时，就会顺便判断能够利用索引字段完成的过滤条件。

只有条件通过的记录，才继续回表。

## 为什么能提升性能

- 减少随机 IO
- 减少回表次数
- 减少 Server 层无效判断

## 适合怎么答

面试里可以直接这样说：

> 索引下推本质上是把部分过滤条件提前到存储引擎层执行，避免先回表再过滤。它主要优化的是二级索引扫描阶段的无效回表问题，尤其在联合索引过滤不完全命中的场景下收益明显。

## 追问

### 不是所有条件都能下推吗

不是。只有存储引擎在索引层就能判断的条件，才有机会下推。

### 什么时候收益最大

当索引命中范围较大、但最终真正满足条件的数据较少时，收益最大。
