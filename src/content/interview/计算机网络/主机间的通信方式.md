##  通信组织模式
主机间通信从组织模式上看，常见的有 C/S 和 P2P。
- C/S 是客户端发送请求、服务器处理并返回结果，特点是集中管理、结构清晰。
- P2P 是节点地位对等，每个节点既可以请求服务，也可以提供服务，特点是去中心化、资源共享能力强。
在 Web 场景下，B/S 一般可以视为 C/S 的一种具体形式。


## 一、C/S 模式的演进
### 1. 单机时代 → 两层 C/S（1980s~1990s）
最早的"客户端/服务器"分层是为了将数据存储从业务逻辑里分离出来。客户端承担全部 UI + 业务逻辑，服务器只做数据存储和查询。这被称为 Fat Client（胖客户端），缺点很明显：业务逻辑变化时，所有客户端都要重新部署。

### 2. 三层 C/S（1990s~2000s）
为了解决胖客户端的维护问题，引入中间层（Application Server），职责拆成三层：表现层（Client）→ 业务逻辑层（App Server）→ 数据层（DB Server）这是现代 Web 应用的直接前身。

### 3. B/S：C/S 的 Web 具现化
B/S 本质上是"以浏览器作为标准化客户端"的三层 C/S。

最大的工程意义在于：客户端零安装、零部署，浏览器成为通用容器。

但早期 B/S（CGI/PHP/JSP 时代）有明显局限：每次交互都刷新整页，用户体验差，服务器压力大。AJAX（2005 年 Jesse James Garrett 命名）改变了这一点——客户端可以局部更新，从此 B/S 真正有能力替代传统富客户端应用。

## 二、现代 C/S 的常规方案
现代系统中，C/S 的"客户端"已经极度多元化：通信协议层面：

- HTTP/REST：无状态、资源导向，最广泛
- gRPC：基于 HTTP/2 + Protobuf，低延迟、强类型，适合服务内部调用
- WebSocket：全双工长连接，适合实时场景（聊天、协同编辑、行情推送）
- GraphQL：客户端按需查询，减少过取/欠取问题

架构模式层面：

- BFF（Backend for Frontend）：为不同前端（Web/移动/TV）定制专属后端聚合层，避免通用 API 在每个端都做大量适配
- API Gateway：统一入口，处理认证、限流、路由，下游是微服务集群

## 三、P2P 模式的演进与分类
P2P 的核心问题是：节点如何发现彼此（节点发现问题）。这直接决定了 P2P 的架构形态：
第一代：集中式索引（Napster，1999）
仍有中央索引服务器记录节点信息，文件传输才是 P2P。本质上是"混合型"——索引集中，数据分散。缺点：中央节点被关停则全盘崩溃。

第二代：纯分散式（Gnutella，2000）
完全去中心，通过洪泛（Flooding）广播查询请求。优点是无单点，缺点是网络流量开销极大，可扩展性差。

第三代：结构化 DHT（Kademlia，BitTorrent，2001~至今）
引入 分布式哈希表（DHT），每个节点负责存储一部分索引，节点查找时间复杂度 O(log N)。这是目前最主流的 P2P 架构基础，BitTorrent、IPFS、以太坊节点发现都在用 Kademlia 的变体。


## 四、P2P 的现代延伸
- 区块链：所有节点对等持有账本副本，共识机制（PoW/PoS）替代中央权威
- IPFS（InterPlanetary File System）：内容寻址 + DHT，URL 指向"内容哈希"而非服务器位置，理论上内容永久可达
- WebRTC：浏览器间直接建立 P2P 媒体通道（视频通话、屏幕共享），但实际上仍需 STUN/TURN 服务器做 NAT 穿透辅助——是典型的"逻辑 P2P，物理上借助辅助服务器"
- BT 种子和磁力链接
